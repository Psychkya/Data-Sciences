Convert the type to a factor and the text to a corpus.

#convert type to factor
hamspam$Type <- as.factor(hamspam$Type)
#convert the text portion to a corpus
hs_corpus <- VCorpus(VectorSource(hamspam$Text))

Perform the necessary cleaning of the corpus

#Clean the corpus
hs_corpus_cln <- tm_map(hs_corpus, content_transformer(tolower))
hs_corpus_cln <- tm_map(hs_corpus_cln, removeNumbers)
hs_corpus_cln <- tm_map(hs_corpus_cln, removeWords, stopwords())
hs_corpus_cln <- tm_map(hs_corpus_cln, removePunctuation)
hs_corpus_cln <- tm_map(hs_corpus_cln, stemDocument)
hs_corpus_cln <- tm_map(hs_corpus_cln, stripWhitespace)

Lets see what a word cloud shows us: There still seems to be a lot of words that does not seem to convey a whole lot of meaning. The words ham and call seems to dominate. We will proceed to create a document term matrix. We will also remove any words occuring less than 5 times.

#Create DTM
hs_DTM <- DocumentTermMatrix(hs_corpus_cln)
#Remove words occuring less than 5 times
freq_words <- findFreqTerms(hs_DTM, 5)
hs_DTM_freq <- hs_DTM[,freq_words]

Since Naive Bayes works on categorical data, we will convert the term counts to YES or NO. If term count is zero, it will be NO otherwise YES.

#Convert counts to YES or NO
conv_counts <- function(x){x <- ifelse(x > 0, "YES", "NO")}
hs_DTM_cat <- apply(hs_DTM_freq, MARGIN = 2, conv_counts)

Normally, Naive Bayes works on a matrix. But to be on the safe side, converted the matrix to a data frame

#convert it to a data frame
hs_df <- as.data.frame(hs_DTM_cat)

We will split the resulting data frame to a test and train set

#split train and test
index <- 1:floor(nrow(hs_DTM)/3)
hs_train <- hs_df[-index,]
hs_test <- hs_df[index,]
hs_train_label <- hamspam$Type[-index]
hs_test_label <- hamspam$Type[index]

We will proceed to build the model and predict the test set

#Build model
hs_model <- naiveBayes(hs_train,hs_train_label)
hs_test_pred <- predict(hs_model,hs_test)

The cross table below shows that the model has an error rate of approx. 2%. For a spam filter, reducing false positive is more important. In this case, false positive error rate is approx. 1.6%.

#Cross table

To find ROC and AUC, we will have to first convert ham and spam to 1 and 0. ham is considered 1, while spam is considered 0.

#ROC
library(ROCR)
## Loading required package: gplots
## 
## Attaching package: 'gplots'

## 
##     textplot
## The following object is masked from 'package:stats':

#In order for ROC to function, need to convert ham and spam to 1 and 0
conv_num <- function(x){x <- ifelse(x=="ham",1,0)}
hs_pred_ROC <- sapply(hs_test_pred, conv_num)
hs_label_ROC <- sapply(hs_test_label, conv_num)
pred <- prediction(hs_pred_ROC,hs_label_ROC)
perf <- performance(pred, measure = "tpr", x.measure = "fpr")

The peformance plot is as below: Finally, we will calculate AUC

auc <- performance(pred,measure = "auc")@y.values[[1]]

The AUC value is 0.9465701 This model performed pretty good. Laplacian correction is added to while calculating the probabilities of the independent attributes just in case for some attribute the probability turns out to be zero. Adding, say 1, to each occurance of an attribute should have practically no impact to the probability distribution of the attribute. Instead, it will prevent the probability of a certain attribute to be zero.

#Try with a small laplacian correction
hs_model_lap <- naiveBayes(hs_train,hs_train_label,laplace = 1)
hs_test_pred_lap <- predict(hs_model_lap,hs_test)

From the cross table below, we see that with laplace correction, the model perfomed slightly worse. The laplace correction can be adjusted to see if makes a difference, but we will not attempt here.
